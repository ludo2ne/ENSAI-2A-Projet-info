Lab 2 - Refactoring a Monolithic Service
Author

Ludovic Deneuville

Before you start

This tutorial combines explanations and code phases. The explanations should not take precedence over those of your teacher.

NoteConcepts covered
  • Refactoring existing code
  • Design Patterns: Factory Pattern & Strategy Pattern
  • Dependency Injection
  • Data Persistence (History tracking)
  • Advanced Unit Testing

You might be thinking: “All this architecture just for a simple coin flip?” … You’re right. And don’t get too comfortable, because we’re about to add a dozen more classes just for fun!” 🙃

1 The Mission 🎯

Currently, our gaming engine only supports a single game mode: Coin Flip.

Our goal is to refactor the system to easily integrate new games, such as Dice, and more complex betting mechanics.

1.1 Current State: A “God Service”

Currently, the GameService class is a Monolith.

It is too heavy because it handles too many responsibilities:

  • Game Logic: It decides the outcome (e.g., random coin flip)
  • Scoring Logic: It manages the mathematical Elo calculation
  • Data Management: It directly interacts with PlayerDao to update scores
  • No History: Once a game is played, the result is lost. It lacks any way to track match history

This design violates the Open/Closed Principle: adding a new game (like “Dice”) or a new scoring rule (like “Flat points”) requires modifying the core GameService code, making it fragile and hard to test.

1.2 Target Architecture: A Modular Engine

Our goal is to transform this monolith into a decoupled system where:

  • The Service is an Orchestrator (it tells others what to do, but doesn’t know how they do it)
  • The Rules are interchangeable (Strategy Pattern)
  • The Creation is automated (Factory Pattern)
  • The History is permanent (Persistence)

2 Modelling and Implementation

2.1 Decoupling Game Logic (Factory Pattern)

We want to move away from if/else blocks to manage different games.

    • Create an abstract base class GameRules with a method play(p1, p2) -> MatchResult.
    • Implement CoinFlipRules (using the existing logic).
    • Implement DiceRules (new logic: roll two dice; if different, one wins; if equal, it’s a draw).
    • Create a GameFactory class with a method get_rules(game_type: str) -> GameRules.

2.2 Decoupling Scoring (Strategy Pattern)

Currently, Elo is hardcoded. We want to be able to switch between “Elo” and “Flat” scoring.

    • Create an abstract class ScoringStrategy with a method compute(p1, p2, winner).
    • Implement EloScoring (move the existing math here).
    • Implement FlatScoring (a simple \(+10 / -10\) points rule).
    • Modify GameService.__init__ to accept a ScoringStrategy.
    • Update play() to use self.scoring_strategy.compute(...).
TipConcept: Dependency Injection

By passing the strategy in the constructor, the GameService no longer decides how to score; it is given a tool to do it. This makes the code much easier to test!

2.3 Data Persistence (History Tracking)

A game is an event that must be recorded.

    • Create a new Business Object GameResult (containing: player1_id, player2_id, game_type, winner_id, timestamp).
    • Create a GameDao to persist these results in a games table.
    • Modify GameService.play() to call GameDao.save(match_result) at the end of the process.

3 Testing & Validation

Your refactoring is successful only if the code is easier to extend and still works perfectly.

    • Test that GameFactory returns the correct rules.
    • Test that DiceRules correctly handles a tie (winner = None).
    • Test that GameService correctly uses the injected ScoringStrategy.
ImportantFinal Checklist
  1. Is GameService.play() free of if/else regarding the game type?
  2. Can I change the scoring mode just by changing the object passed to GameService?
  3. Does the system correctly handle a Draw (no Elo change, but history recorded)?
